# Концепция программного проекта

## Название проекта

**«ИС учёта и обработки заявок IT-поддержки государственного учреждения»**

---

# 1. Описание проблемы

| Пункт | Описание |
|---|---|
| Проблема | В государственных учреждениях обработка заявок в IT-отдел часто ведётся через телефон, мессенджеры или устные обращения. Из-за этого заявки теряются, отсутствует контроль сроков выполнения и сложно отслеживать загрузку IT-специалистов. |
| Воздействует на | Сотрудников учреждения, IT-специалистов, руководство подразделения. |
| Результатом чего является | Увеличение времени решения технических проблем, потеря заявок, отсутствие прозрачности работы IT-подразделения, снижение эффективности сотрудников учреждения. |
| Выигрыш от ПП | Централизованный учёт заявок, контроль статусов выполнения, распределение нагрузки между специалистами, повышение скорости обработки обращений, хранение истории заявок и улучшение качества IT-поддержки. |

---

# 2. Назначение программного средства

## Какие системы смогут применять ПС для решения своих задач?

Программное средство может использоваться:

- IT-подразделениями государственных учреждений;
- отделами технической поддержки организаций;
- административными подразделениями;
- небольшими и средними организациями с внутренней IT-поддержкой.

---

## От каких систем/пользователей поступают входные данные ПС?

### Пользователи системы:

1. Сотрудник учреждения
   - создаёт заявки;
   - отслеживает статус обращения;
   - получает уведомления о выполнении.

2. IT-специалист
   - принимает заявки в работу;
   - изменяет статусы;
   - оставляет комментарии;
   - закрывает обращения.

3. Администратор системы
   - управляет пользователями;
   - назначает роли;
   - контролирует работу системы;
   - просматривает статистику.

---

## Кто/что использует результаты?

- сотрудники учреждения;
- IT-отдел;
- руководство подразделения;
- администрация организации.

## Какие сторонние системы/сервисы участвуют в процессе обработки данных?

- база данных PostgreSQL;
- почтовый сервис уведомлений;
- операционная система сервера;
- веб-браузер пользователя.

# 5. Описание программного продукта по шаблону

Для сотрудников и IT-подразделений государственных учреждений, которым требуется удобный инструмент учёта и обработки технических обращений, данный программный продукт «ИС учёта и обработки заявок IT-поддержки государственного учреждения» является системой автоматизации обработки IT-заявок, которая позволяет централизованно регистрировать, отслеживать и контролировать выполнение обращений сотрудников.

В отличие от устных обращений, телефонных звонков и переписки в мессенджерах, наш продукт обеспечивает прозрачный контроль заявок, хранение истории обращений и повышение эффективности работы IT-подразделения.

# 3. Варианты использования

---

## Use Case №1

### Название:
Создание заявки

### Актор:
Сотрудник учреждения

### Предусловие:
Пользователь авторизован в системе.

### Основной поток:
1. Сотрудник открывает страницу создания заявки.
2. Сотрудник вводит тему обращения.
3. Сотрудник описывает возникшую проблему.
4. Сотрудник выбирает категорию заявки.
5. Сотрудник нажимает кнопку «Создать заявку».
6. Система сохраняет заявку в базе данных.
7. Система присваивает заявке статус «Новая».
8. Система отображает созданную заявку в списке заявок сотрудника.

### Альтернативные потоки:
- Если обязательные поля не заполнены, система выводит сообщение об ошибке.
- Если произошла ошибка сохранения, система сообщает пользователю о невозможности создать заявку.

### Постусловие:
Новая заявка сохранена в системе и доступна для обработки IT-специалистами.

---

## Use Case №2

### Название:
Просмотр статуса заявки

### Актор:
Сотрудник учреждения

### Предусловие:
Сотрудник авторизован в системе и ранее создал хотя бы одну заявку.

### Основной поток:
1. Сотрудник открывает раздел «Мои заявки».
2. Система отображает список заявок сотрудника.
3. Сотрудник выбирает нужную заявку.
4. Система отображает подробную информацию по заявке.
5. Сотрудник просматривает текущий статус заявки и комментарии IT-специалиста.

### Альтернативные потоки:
- Если у сотрудника нет созданных заявок, система отображает пустой список.
- Если заявка не найдена, система выводит сообщение об ошибке.

### Постусловие:
Сотрудник получил информацию о текущем состоянии своей заявки.

---

## Use Case №3

### Название:
Обработка заявки

### Актор:
IT-специалист

### Предусловие:
IT-специалист авторизован в системе. В системе существует хотя бы одна заявка со статусом «Новая» или «В работе».

### Основной поток:
1. IT-специалист открывает список заявок.
2. Система отображает доступные заявки.
3. IT-специалист выбирает заявку.
4. IT-специалист назначает заявку на себя.
5. Система изменяет ответственного исполнителя заявки.
6. IT-специалист меняет статус заявки на «В работе».
7. IT-специалист выполняет необходимые действия для решения проблемы.
8. IT-специалист добавляет комментарий о выполненной работе.
9. IT-специалист меняет статус заявки на «Выполнена» или «Закрыта».
10. Система сохраняет изменения.

### Альтернативные потоки:
- Если заявка уже назначена другому специалисту, система выводит уведомление.
- Если комментарий не добавлен при закрытии заявки, система может запросить пояснение.
- Если заявка не может быть обработана, IT-специалист оставляет комментарий с причиной.

### Постусловие:
Заявка обработана, её статус изменён, комментарий сохранён в системе.

---

## Use Case №4

### Название:
Управление пользователями

### Актор:
Администратор системы

### Предусловие:
Администратор авторизован в системе.

### Основной поток:
1. Администратор открывает раздел управления пользователями.
2. Система отображает список зарегистрированных пользователей.
3. Администратор создаёт нового пользователя или выбирает существующего.
4. Администратор указывает данные пользователя.
5. Администратор назначает пользователю роль.
6. Администратор сохраняет изменения.
7. Система обновляет данные пользователя в базе данных.

### Альтернативные потоки:
- Если пользователь с таким логином уже существует, система выводит сообщение об ошибке.
- Если обязательные поля не заполнены, система предлагает заполнить недостающие данные.
- Если роль не выбрана, система не позволяет сохранить пользователя.

### Постусловие:
Данные пользователя сохранены, роль пользователя определена.

---

## Use Case №5

### Название:
Просмотр статистики и отчётов

### Актор:
Администратор системы

### Предусловие:
Администратор авторизован в системе. В системе имеются созданные или обработанные заявки.

### Основной поток:
1. Администратор открывает раздел статистики.
2. Система отображает общую информацию по заявкам.
3. Администратор выбирает период для формирования статистики.
4. Система формирует отчёт по заявкам.
5. Администратор просматривает количество новых, выполненных и закрытых заявок.
6. Администратор анализирует загрузку IT-специалистов.

### Альтернативные потоки:
- Если за выбранный период нет данных, система выводит уведомление.
- Если отчёт не может быть сформирован, система сообщает об ошибке.

### Постусловие:
Администратор получил статистику по работе системы и IT-подразделения.

---

## Use Case №6

### Название:
Просмотр журнала действий

### Актор:
Администратор системы

### Предусловие:
Администратор авторизован в системе.

### Основной поток:
1. Администратор открывает журнал действий.
2. Система отображает список действий пользователей.
3. Администратор фильтрует записи по пользователю, дате или типу действия.
4. Система отображает подходящие записи.
5. Администратор просматривает информацию о действиях в системе.

### Альтернативные потоки:
- Если записей по выбранным параметрам нет, система выводит пустой список.
- Если данные журнала недоступны, система сообщает об ошибке.

### Постусловие:
Администратор получил информацию о действиях пользователей в системе.
